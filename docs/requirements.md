## 1. 概要と目的

### 1.1 背景

日本人(あなた)と外国人パートナー(彼女)が、Discord 上で日常的に会話している。あなたは英語を、彼女は日本語をそれぞれ学習中で、会話の最中にわからない単語・表現が出てきた際、Discord の外で辞書や翻訳サイトを開いて調べる必要がある。これは会話の流れを止め、学習のモチベーションも下げる。

### 1.2 ゴール

- Discord 内で会話を中断せずに、単語・文章・文法の解説を受けられる状態を作る。
- 学習者の母語に合わせた解説で、効率的にインプットできるようにする。
- 調べた単語を蓄積し、週次でまとめて復習できるようにする。
- 2 人が同時に互いの学習を支援できる(自分が相手のために調べることも自然にできる)。

### 1.3 非ゴール(本プロジェクトでは扱わない)

- 音声・発音機能(将来検討)。
- 単語帳アプリのような長期 SRS(間隔反復)学習機能。
- 他言語(英語・日本語以外)への対応。
- 一般ユーザー(自分達 2 人以外)向けのサービス化。

---

## 2. ユーザー

### 2.1 ペルソナ A: 英語学習者(あなた)

- 母語: 日本語
- 学習言語: 英語
- 主なユースケース:
  - 彼女から英語メッセージを受け取り、わからない単語を調べる。
  - 彼女に伝えたい日本語の単語を、彼女向けに英語解説付きで共有する。

### 2.2 ペルソナ B: 日本語学習者(彼女)

- 母語: 英語(または他言語、UI 上は英語で対応)
- 学習言語: 日本語
- 主なユースケース:
  - あなたから日本語メッセージを受け取り、わからない単語を調べる。
  - あなたに伝えたい英語の単語を、あなた向けに日本語解説付きで共有する。

### 2.3 利用シーン

会話中に「これってどういう意味?」と思った瞬間に、Bot へメンションして即座に解説を得る。学習中の言語の Bot を呼ぶことで、自分のための調べ物にも、相手への教材提供にもなる。

## 3. スコープ

### 3.1 含むもの

- Discord 上での単語・文章・文法に関する自動回答。
- 英↔日 双方向の解説。
- 検索履歴の保存と週次レポート配信。
- 辞書ページへのリンク提供。

### 3.2 含まないもの

- 外部公開・他サーバーへの導入。
- Web UI、モバイルアプリ、メール通知などの Discord 外配信。
- 音声入出力。
- 課金ユーザー管理。

## 4. 機能要件

### 4.1 単語翻訳機能

- 入力: 単語 1 つ / 短いフレーズ(例: `apple`, `林檎`, `pick up`)。
- 出力(複数語義対応): 1〜3 個の「語義(sense)」を返す。各語義は以下を持つ:
  - 見出し語(`headword`): 常に target_lang の語(逆引き時も target_lang を主見出しに据える)
  - 振り仮名(`headword_reading`): 日本語先生Bot で見出し語に漢字を含む場合のみ。それ以外は空
  - 品詞(`part_of_speech`)
  - 訳語(`translations`): explanation_lang の語/フレーズ 1〜3 個(典型順)
  - 例文: ちょうど 2 件(対訳付き)
- 語義が 1 つのみの一般語(例: `apple`)は 1 語義で返す。複数の明確な意味を持つ語(例: `bank` → 銀行 / 土手)は語義を分割する。
- 辞書ページへのリンク(語全体に対して 1 つ)。
- 順引き / 逆引き(MODE A / B): 入力が target_lang の語なら順引き(MODE A、全語義が同じ見出し語)、入力が explanation_lang の語なら逆引き(MODE B、語義ごとに別の target_lang 見出し語)。判定は LLM がプロンプト内で行う。詳細は [bot-spec.md](./bot-spec.md) の「発火条件」「逆引きユースケース」を参照。
- 「意味の文章説明」「使い方(コロケーション)」フィールドは持たない(訳語 + 例文で代替)。
- 利用 LLM: フォールバックチェーン([architecture.md](./architecture.md) の「LLM フォールバックチェーン」参照)。一次は `gemini-3.1-flash-lite`。

### 4.2 文章翻訳機能

- 入力: 複数語からなる文(例: `Could you pick me up at the station?`)。
- 出力:
  - 自然な訳文
  - 直訳(必要に応じて、ニュアンス理解の補助。不要なら空)
  - 重要表現の補足(0〜3 個)
  - 振り仮名(`source_reading`): 日本語先生Bot で原文に漢字を含む場合のみ
- 文章も単語同様に逆引き対応(入力が母語の文なら target_lang に翻訳したものを原文として扱う)。
- 利用 LLM: フォールバックチェーン([architecture.md](./architecture.md) の「LLM フォールバックチェーン」参照)。

### 4.3 文法解説機能

- 入力: 文法事項に関する質問(例: `would have p.p.の使い方`, `〜てしまう の意味`)。
- 出力:
  - 文法ルールの解説(トピック名 + 解説本文)
  - 例文 2〜3 件(対訳付き)
  - 類似文法との違い(あれば。なければ空)
- 利用 LLM: フォールバックチェーン([architecture.md](./architecture.md) の「LLM フォールバックチェーン」参照)。

### 4.4 入力種別の自動判定

- 判定方式: **LLM による自動判定**(`src/handlers/router.py` が分類専用のシステムプロンプトで LLM を 1 回呼ぶ)。各ハンドラ呼び出しとは別の、先行する LLM コール。
- 判定対象: 「単語(word)」「文章(sentence)」「文法質問(grammar)」の 3 種。
- 失敗時の挙動: LLM が 3 種以外を返した場合は「単語(word)」にフォールバックし警告ログを出す(フォロー質問は付けない)。

### 4.5 Web リンク挿入

- リンク先:
  - 英語の単語: Cambridge Dictionary (`https://dictionary.cambridge.org/dictionary/english/{word}`)
  - 日本語の単語: Jisho.org (`https://jisho.org/search/{word}`)
- 実装方式: 単語文字列を URL エンコードしてリンク生成(API 呼び出し不要)。
- リンク到達性の確認: 行わない(404 でもユーザーが他で検索できれば実用上問題なしと判断)。

### 4.6 クエリログの記録

- 記録対象: **単語・文章・文法の全クエリ**。
- 記録項目:
  - 種別(`kind`: word / sentence / grammar)
  - 検索した Discord ユーザー ID(と表示名)
  - 対象言語(`target_lang`: en / ja)
  - クエリ本文(`query_text`)
  - 結果サマリ(`result_summary`): word は全語義の見出し語を `/` 連結、sentence は自然訳、grammar はトピック名
  - 振り仮名(`reading`): 日本語の見出し語/原文に漢字を含む場合の読み(週次レポート表示用)。なければ空
  - 検索日時(JST)

### 4.7 週次レポート

- タイミング: 毎週日曜 21:00 JST。
- 投稿先: 専用チャンネル(`.env` の `REPORT_CHANNEL_ID` で指定)。
- 形式: 1 学習方向(`target_lang`)につき 1 つの Embed。中に 3 セクション(📘 単語 / 📝 文章 / 📚 文法)。
- **集計単位は target_lang のみ**:「誰が調べたか」は無視し、target_lang ベースで学習者にひも付ける。
  - 例: Chris が `視察` を調べても、`視察` は日本語学習者(Camille)のレポートに入る
  - 理由: 復習対象は「学んでいる言語の語彙」であり、querier は無関係
- 学習者名は `.env` の `EN_LEARNER_NAME` / `JA_LEARNER_NAME` で指定。
- 集計範囲: 今週(月曜 0:00 〜 実行時刻、JST)。
- 重複: 同じクエリが複数回検索されても 1 行にまとめ、検索回数を併記。
- 0 件の target_lang / kind は表示しない(ノイズ抑制)。
- 詳細仕様は [bot-spec.md](./bot-spec.md) の「週次レポート仕様」を参照。

### 4.8 デイリークイズ機能

- **配信タイミング**: 毎朝 8:00 JST(APScheduler の cron)。
- **配信単位**: 学習者ごとに個別問題(EN 学習者は英語、JA 学習者は日本語)。
- **1 日の問題数**: 2 問(復習 1 + 新出 1、両方クイズ形式)。
- **問題種別**: word のみ。
- **出題形式**: 4 択ボタン(Discord Button Component)。
- **復習ソース**: `query_log` 履歴から、過去その学習者が調べた語を選定。
- **新出ソース**: LLM 1 回コールで「学習者の直近履歴 + 過去出題除外リスト」を渡し、同レベル相当の未学習語を提案。
- **回答権限**: 該当学習者本人のみ。他者のボタン押下はエフェメラル応答で弾く。
- **正解発表**: 本人がボタンを押した瞬間に即時(不正解でも一発で正解+解説を表示)。
- **未回答時**: 放置(自動で答えを出さない、統計用に `answered_at` NULL のまま残す)。
- **追加クイズ(おかわり)**: その日の配信分を全て回答し終えた本人に「もっとやる?」と提示し、+1/+2/+3 の新出クイズを 1 日 1 回まで追加できる(詳細は [bot-spec.md](./bot-spec.md) の「追加クイズ(おかわり)」)。
- **キャラクター付与**: なし。事典トーンで統一。

詳細仕様は [bot-spec.md](./bot-spec.md) の「デイリークイズ仕様」を参照。

## 5. 非機能要件

### 5.1 可用性

- AWS EC2(t3.micro / ap-northeast-1)上で Docker Compose により **24 時間稼働**。コンテナは `restart: unless-stopped` で自動再起動する。
- prod / dev を同一 EC2 上に並存(2 環境とも常時稼働)。EC2 ホスト障害時は両環境が同時に停止する(冗長化なし)。
- メッセージキューは持たない。デプロイ・再起動等で一時的に Bot が落ちている間のメンションは処理されない(再送が必要)。
- コストガードにより、推定請求額が $2 を超えると課金アラームが EC2 を自動停止する。発火時は手動で再起動するまで応答しない([aws-infrastructure.md](./aws-infrastructure.md) の 2.11 参照)。
- インフラ構成の詳細は [aws-infrastructure.md](./aws-infrastructure.md) を参照。

### 5.2 レスポンス時間目標

| 機能     | 目標応答時間 |
| -------- | ------------ |
| 単語翻訳 | 3 秒以内     |
| 文章翻訳 | 3 秒以内     |
| 文法解説 | 5 秒以内     |

通常時は一次モデル `gemini-3.1-flash-lite` で応答。ただし 1 リクエストにつき分類用 + ハンドラ用で LLM を 2 回直列に呼ぶため、レート制限ヒット時のフォールバック([architecture.md](./architecture.md) の「LLM フォールバックチェーン」)発生時は上記目標を超える場合がある。

応答に時間がかかる場合は、Discord の「考え中」インジケータ(typing indicator)で UX を維持する。

### 5.3 コスト

- LLM(Gemini / OpenRouter)はいずれも無料枠内で運用する前提のため、API 利用料は $0。
- AWS インフラ(EC2 t3.micro + EBS gp3 8 GiB、prod/dev 兼用)に課金が発生し得るが、$2 の課金アラームでEC2 を自動停止する ため AWS 請求は実質 $2 程度で頭打ち(無料枠が効いている間はほぼ $0)。詳細は [aws-infrastructure.md](./aws-infrastructure.md) の 2.11 を参照。
- NAT Gateway・RDS・CloudWatch Logs・Secrets Manager は不使用のため追加費用なし。
- 正確な金額は AWS の請求情報で確認する。
- 想定超過時の対応: 5.5 を参照。

### 5.4 プライバシー

- 会話履歴は Discord 上にのみ存在。Bot は検索・クイズのログ以外を保存しない。
- 検索ログ DB(SQLite)は運用者自身の AWS アカウント内の EC2(EBS)に保存し、第三者サービスへは送信しない。
- ユーザーが Bot に送信した内容(単語・文章・文法質問)は LLM API(Gemini、フォールバック時は OpenRouter)に送られる前提を受け入れる。機微情報は Bot に送らない運用とする。

### 5.5 制約・フェイルセーフ

- LLM 失敗時: フォールバックチェーン([architecture.md](./architecture.md) の「LLM フォールバックチェーン」)を順に試す。レート超過(429)/モデル不在(404)/サーバ過負荷(5xx)は次段へ自動フォールバック。それ以外の例外は即座に伝播。
- 全段(Gemini 一次 → Gemini 二次 → OpenRouter)を使い切った場合: `on_message` 側で例外を捕捉し、エラーメッセージを Discord 上に返す(例: "ごめん、エラーが出ました: ...")。
- 応答した実モデル/プロバイダは Embed フッターに `via {モデル名}` として表示する。

## 6. 制約事項・前提

### 6.1 運用前提

- EC2 インスタンスと Bot コンテナが稼働している時のみ機能する。
- LLM API(Gemini、フォールバック時は OpenRouter)の到達性を前提とする(到達できない場合は応答できない)。

### 6.2 外部依存

- Discord API のレート制限(基本的に問題にならない規模だが、メッセージ送信は 5 req/sec 程度を目安に)。
- LLM API(Gemini / OpenRouter)の到達性に依存する。Gemini の無料枠超過(429)・モデル不在(404)・サーバ過負荷(5xx)はフォールバックチェーンで吸収し、全段が枯渇した時のみエラーメッセージを返す(5.5 参照)。

### 6.3 セキュリティ

- Discord Bot トークン・Gemini / OpenRouter API キー等のシークレットは AWS SSM Parameter Store(SecureString)で管理する(prefix `/language-teacher/{prod,dev}/*`)。
- デプロイ時に `scripts/deploy.sh` が SSM から値を取得し、EC2 上に `.env.en` / `.env.ja` を生成する(`.env` はリポジトリに含めず `.gitignore` で除外)。
